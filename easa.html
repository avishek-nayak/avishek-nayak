Problem Statement

EASA is a global authority in the electrical motor database industry. Its flagship calculation tool, currently on version V4, is available only as a native Windows application, which limits accessibility and prevents new engineers from trying the product before purchase.

Beyond access, the tool carries a steep learning curve. Every new engineer requires three months of intensive, mandatory training delivered by Eugene Vogel, a technical EASA engineer whose role is to onboard users into the software. This dependency creates a bottleneck for adoption and slows time-to-productivity for new hires.

The CEO's vision is twofold: make the tool easier to access across platforms, and grow the user base so younger engineers can experience the product earlier in their careers and convert into paying users over time.

The redesign operates under one critical constraint. The existing user base is predominantly in the 50+ age group, and these power users will remain the core revenue drivers for the next several years until the share of younger engineers grows. As a result, the redesign must retain complete functional parity with V4, preserving the workflows, logic, and depth that long-time users rely on, while modernizing the experience for newcomers.

Training was a symptom, not a solution

The 3-month onboarding wasn't a knowledge gap — it was an interface gap. The complexity of the tool had been transferred entirely to the user. Tesler's Law in plain sight.

The desktop app's pinnable nested tabs were a Windows-specific convention. Replicating them in a browser viewport was technically and experientially impossible.

03

Key Constraints

The decisions that shaped every screen

Three constraints defined the architecture of the entire platform. Each one required negotiation, not just execution.

The no-scroll demand

Client Constraint → Design Negotiation

The client's initial position was firm: all data must be visible in the viewport with no scrolling — matching the feel of the desktop application. This was pushed back on immediately.

After deeper sessions, the real concern surfaced: they were afraid important data would be hidden or forgotten below the fold. That was a UX problem, not a technical requirement.

Resolution

Established which data was truly primary. Demonstrated how browser zoom and resolution behavior works. Made the case that white space reduces cognitive load — it is not wasted space. Compromise reached: critical data stays above the fold. Secondary data can scroll.

Nested tabs in a browser viewport

Technical Constraint → Structural Solution

The desktop app allowed tabs to be pinned and deeply nested. A browser has a fixed viewport — direct replication was not feasible.

Resolution — Conditional Tab Model

Top navigation layer handles primary context switching. A secondary tab at the bottom of the viewport appears only when output is generated — conditional, triggered by user action, not always present.

Retrospective

In hindsight, a split-screen model would have been more intuitive — once output is generated, the screen divides equally between input and result. The bottom tab works, but requires the user to understand its conditionality. A split screen makes it self-evident. This is what I would design differently today.

The 1,000 motor filter constraint

System Constraint → UX Pattern

The database held an enormous volume of motor records. The system could not display results above 1,000 motors — querying without filtering would return data that couldn't render or be meaningfully navigated.

Resolution — Progressive Narrowing with Live Feedback

Real-time result count shown throughout filtering — guiding users toward a manageable set without a hard error block. As the count approached the threshold, users felt momentum, not restriction. The filter was designed as a collaborative narrowing process, not a gatekeeper.

Design System — Built from Scratch

Proprietary design system in Figma covering color tokens, typography scale, spacing system, component states, and role-based variants. Built specifically for the edge cases of a data-dense industrial SaaS — 20-column tables, compact card views, conditional UI states.

04

Developer Collaboration

Backend sessions clarified data structure and filter logic. Frontend sessions resolved component-level feasibility — including a mid-project keyboard accessibility rework that required restructuring certain components into single focusable containers for logical tab order.

05

Usability Reviews

Informal reviews with Eugene at key structural points — validating whether layouts and navigation made conceptual sense before committing to UI execution. All design decisions were locked at wireframe stage before any visual design began.

Key Design Decisions

Where the thinking happened

Card view vs. Table view — The 20-column problem

Motor data tables ran to 20+ columns. Both views were required — but a card view with internal scroll dramatically increases cognitive load for older users. The solution: a compact card with a fixed, curated set of critical columns for identification and comparison. Full table view available as a toggle for power users.

[ Insert: Card view vs. table view toggle — motor listing screen ]

Designing for an older user base

The primary users were experienced, senior engineers. Every spatial, sizing, and grouping decision was made with this in mind — generous touch targets, logical keyboard navigation, WCAG-compliant contrast across all data and number-heavy screens.

Mobile adaptation — drawing the line

Adaptive rather than fully responsive. Structure and navigation adapted to mobile — but data-intensive views remained desktop-first.

Senior Design Insight

Some features are non-negotiable. Business decisions sometimes require the designer to find the best way to present something, not to question whether it should be there. Knowing when to advocate for the user and when to align with the business is a judgment call — not a rule. Recognising it, naming it, and communicating it to stakeholders is part of the design work.

Laws applied as design decisions — each principle drove a specific choice, shaped a trade-off, or resolved a constraint.

Hick's Law

Designed for the highest role, restricted downward

Fewer visible choices at each role level = faster decision-making. The interface was never stripped down — it was appropriately scoped.

08

Reflection

"The hardest design work on EASA wasn't drawing screens. It was thinking through every possible use case before a single line hit the canvas — from font height to scroll behaviour, from viewport fitting to convincing a client that white space is doing real work."

The one thing I would design differently: the tab management structure. The conditional bottom tab works — but it requires the user to understand its conditionality. A split-screen model, where generating output divides the screen equally between input and result, would have made that relationship spatially obvious from the first use. That is the growth this project produced.
